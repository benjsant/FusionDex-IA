## Draft principal

> 🧬 **Building in public : FusionDex-IA**
>
> Je construis un Pokédex complet pour **Pokémon Infinite Fusion** — un fan-game où l'on fusionne n'importe quel Pokémon avec n'importe quel autre. ~176 000 combinaisons possibles, aucune API officielle, données éparpillées sur 3 wikis (Wiki IF, PokeAPI, Poképédia).
>
> Le projet me sert à démontrer une stack complète data / backend / frontend / infra sur un domaine que j'aime, et surtout à **livrer en continu** — pas un POC abandonné au bout d'une semaine.
>
> **Où j'en suis aujourd'hui :**
> → ETL Python (uv) qui agrège les trois sources → **572 Pokémon, 676 moves, 166 k sprites de fusion, 65 règles "Move Experts"** chargés en base
> → API FastAPI : **30+ endpoints, 53 tests pytest verts**, schéma bilingue EN/FR
> → Frontend Next.js 15 (App Router) 🚧
> → Stack **100 % Docker** : db, backend, sprites (nginx), frontend, doc MkDocs — dev + override prod avec seul le frontend exposé publiquement
>
> **Cette semaine, j'ai livré (entre autres) :**
>
> 🔸 **La mécanique Move Experts** — deux NPCs enseignent des moves selon ce que la fusion contient (contraintes multi-valeurs sur pokémon / types / moves). Modélisé avec des `INTEGER[]` Postgres plutôt qu'une table de jonction : `&&` et `<@` rendent la requête triviale, tableau vide = axe non contraint. 65 règles parsées depuis un wiki avec des tables `rowspan`.
>
> 🔸 **Le proxy Next.js pour masquer le backend** — un route handler catch-all `/app/api/[...path]/route.ts` lit `process.env` à chaque requête, au lieu d'un `next.config.ts` rewrite qui fige la cible au build. Résultat : **zéro URL backend dans le bundle client**, et on peut changer la cible sans rebuild.
>
> 🔸 **La conteneurisation de la doc** — `docker compose --profile docs up docs` lance MkDocs Material avec hot-reload via volumes. Cohérent avec le principe "tout tourne en Docker" que je m'impose sur ce projet.
>
> Honnêtement : le frontend a encore des pages à polir, l'intégration DeepSeek (couche IA) attend une vraie clé pour validation end-to-end, et le dump SQL fixture pour passer la CI en full pytest reste à committer.
>
> Mais le socle tient, et les ajouts récents (Move Experts + refactor infra) n'ont rien cassé — c'est souvent le vrai test d'une archi.

## Pourquoi ce cadrage

- **Hook technique** (176 k combinaisons, 3 wikis) plutôt que hype marketing.
- **"Je construis"** + **"livrer en continu"** : signale la discipline à un recruteur sans dire "je cherche un job".
- **"Cette semaine"** : montre que le projet est actif aujourd'hui, pas figé. Particulièrement important si le recruteur check le repo — les derniers commits doivent corroborer le post.
- **3 décisions** : chacune est un mini-exercice de raisonnement technique (pourquoi `INTEGER[]`, pourquoi un proxy, pourquoi Docker partout). Deux autres (`CORS defense in depth`, convention ports préfixe-5) sont gardées pour un 2ᵉ post à la v1.0.
- **Honnêteté explicite** sur ce qui reste à faire : crédibilise tout le reste. Un recruteur expérimenté repère un post qui survend.
- **CTA soft** : "commentaires / DM ouverts" — disponibilité sans insistance.
- **Hashtags** : stack + `BuildingInPublic` pour capter les deux audiences (tech + community).

## Variantes à considérer

- **Carrousel** (post multi-images) : découper les 3 décisions en 3 slides avec le code/diagramme de chaque. Plus d'engagement sur LinkedIn mais demande du visuel propre.
- **Post plus court (≈ 150 mots)** : garder juste 1 décision technique + lien. Utile si tu postes régulièrement (2ᵉ post dans la foulée 3-4 jours plus tard avec une autre décision).
- **Ajouter une capture d'écran** (frontend ou Swagger) en image. Même un frontend `🚧` rend mieux qu'un post texte brut — à condition que l'UI ne soit pas gênante.
